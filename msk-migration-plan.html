消费者 offset 切换 · 迁移运行手册

MSK Standard → Express 迁移
消费者 offset 切换的两种方案

MSK 加密配置无法满足 Replicator 要求,改用 MirrorMaker 2(单向 老→新)迁移;消费者的 offset 存在外部数据库,自己 seek。MM2 不保留 offset 值,因此 DB 里的老 offset 在新集群上无效,必须在切换时重新定位。

迁移工具MirrorMaker 2 复制方向单向 · 老 → 新 offset 存储外部数据库 切换方式消费者自行 seek 验证真机 11/11 · 单元 10/10

生产者停机时间⇄实现复杂度
A timestamp 映射脚本生产者基本不停(切换瞬间暂停几秒)
生产者停机 基本不停
实现复杂度 高

消费者按时间戳独立定位,不依赖冻结边界,可带 backlog 切。代价:脚本复杂 + 消费者需幂等去重。

两种方案都不改消费者代码(消费者永远是「读 DB offset → seek」),区别只在于由谁、在何时、用什么逻辑把正确的新集群 offset 写进 DB。

Atimestamp 映射脚本(生产者基本不停)

适用生产者不能长时间停 / 消费者 backlog 大、不想等排空。

原理用时间戳反查新 offset

MM2 复制时保留原始消息时间戳。脚本用消费者在老集群"最后处理完那条消息"的时间戳,在新集群上 offsetsForTimes(ts) 反查出对应的新 offset,写回 DB。因为定位靠时间戳,不依赖新集群末尾是否冻结——即使生产者已经切过去、新集群在持续写入,也能算出正确的 resume 点。

步骤切换流程(每个 topic 独立执行)

  1. 前置:保持 MM2 低延迟(lag 接近 0),让新集群随时几乎追平。新 topic 用 IdentityReplicationPolicy 保持同名。
  2. 生产者短暂排空在途请求(几秒),切 bootstrap.servers + 认证到新集群 :9098 / IAM,立即恢复写入新集群。
  3. 生产者切换后,等 MM2 把老集群最后的尾部复制完(因生产已停老集群,几秒即追平),然后停掉该 topic 的 MM2(从复制列表移除),避免新写入与滞后副本交错。
  4. 跑 timestamp 翻译脚本:对该 topic 每个分区,取 DB 里记录的"最后处理完那条"的时间戳 → 在新集群 offsetsForTimes(ts) → 得新 offset → 覆盖写回 DB。
  5. 切消费者到新集群:照常「读 DB offset → seek」,从翻译后的位置继续消费 backlog,再平滑过渡到新生产的数据。
  6. resume 用 offsets_for_times(ts)(非 ts+1)落在第一条 ts≥最后处理时间戳的消息,即最后处理那条本身——重处理(同时间戳那一组),绝不跳过未处理消息;消费端按业务主键幂等去重吸收这点重复。
为什么生产者能不停 消费者 resume 点由"最后处理消息的时间戳"算出,与新集群是否冻结、是否已有新数据无关。所以生产者切换不需要等消费者排空,也不需要冻结窗口。
前提条件 ① 新 topic 时间戳类型须为默认 CreateTime(否则时间戳被改写,反查失效);② 消费者必须幂等;③ 老集群在跑脚本时仍在线、消息仍在 retention 内。

offset 翻译原理 · 两方案差异的本质

BendOffsets 快照(必须停生产者)

适用能接受生产者短暂停机 / backlog 小、想要最简实现、无重复消费。

原理制造干净边界 E′

制造一个干净边界:停生产者 → 让消费者把老集群消费到 lag=0 → MM2 把老集群全量复制到新集群并排空。此时新集群冻结在末尾 E′,且 E′ 之前的数据消费者都已处理。于是消费者只需从 E′ 开始即可——脚本退化成"取新集群 endOffsets 写进 DB"一行逻辑,无需时间戳、无重复消费。

步骤切换流程(每个 topic 独立执行)

  1. 停老集群该 topic 的生产者。老集群末尾 E 从此不再增长。
  2. 消费者把老集群消费到 lag=0(处理完所有 backlog)。
  3. 等 MM2 排空(老→新 lag=0):新集群拿到老集群 100% 数据,冻结在 E′。
  4. 在冻结窗口内抓 endOffsets=E′,写进 DB(覆盖该分区 offset)。必须在生产者切回之前做,否则 E′ 会被新写入推高 → 跳过新数据。
  5. 切消费者到新集群:读 DB(=E′)→ seek 到 E′,等待新生产。
  6. 停 MM2 → 切生产者到新集群开始写入。消费者从 E′ 顺势接住所有新生产,不漏不重。
顺序是关键(load-bearing) 抓 E′ + 切消费者必须在切生产者之前。若生产者已切、新集群在写,endOffsets 是移动目标,写进 DB 会让消费者从更靠后的位置开始,跳过中间已写入的新消息(丢数据)。
极简脚本 for each partition: db.set(p, newConsumer.endOffsets(tp)) —— 不碰老集群、不用时间戳、无边界特例、无重复消费。代价仅仅是生产者停机时长 = 消费者 drain + MM2 drain 的全程(backlog 越大停越久)。

实测验证结果 · 真实 Amazon MSK 端到端验证

在 us-east-1 的真实 Amazon MSK 集群(2× kafka.t3.small,Kafka 3.6.0,PLAINTEXT)上,用真实 MirrorMaker 2 把 orders 复制到 src.orders(DefaultReplicationPolicy),端到端验证两个脚本。为让测试有意义,复制前给目标 topic 预埋 junk(p0=5、p1=3,旧时间戳),强制目标 offset 与源错位(src.orders p0 末尾=15、p1=13)——这样"直接照搬 offset"的错误实现会失败。时间戳保留已验证:源 orders p0@offset 0 的 k0 → 目标 src.orders p0@offset 5,时间戳完全相同。

评审发现并修复的真 bug 最初用 offsets_for_times(last_msg_ts + 1),在"同一毫秒多条消息"时会静默跳过未处理消息(丢数据)。已改为 offsets_for_times(last_msg_ts)——落在第一条 ts≥last_msg_ts 的消息,重处理(去重兜底)而非跳过;单测 + 真机都验证了跳过安全(场景 S-dup)。

注意事项与要求

  • 目标 topic 必须是 CreateTime(不能 LogAppendTime):否则目标 broker 会用"复制时刻"覆盖 MM2 带来的原始时间戳,方案 A 的时间戳锚点失效。源 topic 哪种都行(只决定时间戳的含义/单调性);MSK/Kafka 默认即 CreateTime。
  • 消费者必须按业务主键幂等去重:方案 A 在边界会重处理"最后处理那条 / 同时间戳那一组",靠去重兜底(本就是 at-least-once 语义)。
  • 生产时间乱序的影响:正常 cutover(最后处理的"锚点"消息仍在目标 retention 内)不会丢数据,最多多重复几条;只有"锚点已被 retention/compaction 删除 + 时间戳乱序"这个极端组合才可能跳过。缓解:保证 retention 覆盖迁移窗口 + 去重,或用方案 B(与顺序完全无关)。
  • MM2 配置:同集群 src==dst 演示时必须关掉 emit.heartbeats(否则 DefaultReplicationPolicy 会把 heartbeats 递归复制成 src.src.…heartbeats 级联,撑爆 topic 名);真实跨集群迁移无此问题。要目标 topic 同名用 IdentityReplicationPolicy。
  • DB 里的老 offset 必须在 cutover 时被覆盖:MM2 不保留 offset 值,老 offset 在新集群无意义。
  • 认证:验证走 PLAINTEXT(9092);生产 IAM 用 --security-protocol SASL_SSL + OAUTHBEARER(aws-msk-iam-sasl-signer)。
  • 为什么用 MM2 而非 MSK Replicator:Replicator 要求源集群开 IAM access control(强制 broker 间加密=开,而该设置创建后不可改);加密关着且不可改时,只能用 MM2。
  • 脚本名:方案 A = migrate_by_timestamp.py;方案 B = migrate_by_endoffsets.py。两者都不改消费者代码。

两方案通用的注意事项

  • 按 topic / 业务分批切,每个 topic 独立做一次小型 cutover,风险隔离、窗口短。切勿把同一个 topic 的生产者拆成"一部分写老、一部分写新"——MM2 单向复制会导致新集群乱序。
  • MM2 用 IdentityReplicationPolicy,让复制后的 topic 在新集群同名,迁移对生产者/消费者透明。
  • 切完的 topic 从 MM2 复制列表移除,只继续复制未迁移的 topic,避免重复/回环。
  • DB 里的老 offset 必须被覆盖:MM2 不保证 offset 相同,老 offset 在新集群是错误数字,直接 seek 会乱读。
  • 有数据依赖的 topic(消费 A 后又生产到 B)应同批或按依赖顺序迁移。
  • 长期增强(可选):给 DB 加一列 last_msg_timestamp,消费者每次提交 offset 时一并存"最后处理那条"的时间戳。这样以后任何跨集群迁移/容灾都能直接用方案 A,省掉到老集群反查时间戳的步骤。

数据OffsetDB 存储样例(SQLite · 每分区一行)

消费者把「下一条要读的 offset(committed_offset)」+「最后处理那条消息的时间戳(last_msg_ts)」存进外部 DB,并用 cluster_id 区分新老集群。迁移脚本只新增「新集群」的行(老行保留,可回滚),消费者代码不动。下面是 schema 与样例(offset 值取自实测 S1/S2):

用 cluster_id 区分,topic 名两边相同(IdentityReplicationPolicy),老/新行共存。migrate_by_timestamp.py 用 last_msg_ts 在新集群 offsetsForTimes 反查、写入 new 行:p0 6 → 10、p1 10 → 12(正是实测 S1 / S2)。建表:offsets(cluster_id, topic, partition, committed_offset, last_msg_ts),主键 (cluster_id, topic, partition)。cutover = 消费者把 cluster_id 从 old 翻到 new(连同 bootstrap),topic 名不变。

消费消费者代码样例(迁移前后完全不变)

# 消费者:迁移前后代码一行都不用改 —— 永远是「读 DB offset → seek」
# cutover 时只把 CLUSTER_ID 从 'old' 翻到 'new'(连同 bootstrap),topic 名和代码都不变
from kafka import KafkaConsumer, TopicPartition
from offset_migration.offsetdb import OffsetDB

CLUSTER_ID = config.CLUSTER_ID            # 'old' / 'new',来自配置(与 bootstrap 一起翻)
db = OffsetDB("offsets.db")
consumer = KafkaConsumer(bootstrap_servers=BOOTSTRAP,
                         enable_auto_commit=False, group_id=None)
tp = TopicPartition(TOPIC, PARTITION)
consumer.assign([tp])

row = db.get(CLUSTER_ID, TOPIC, PARTITION)   # -> (committed_offset, last_msg_ts)
consumer.seek(tp, row[0] if row else 0)      # 从 DB 记录的位置继续

for rec in consumer:
    process(rec)                          # 业务处理:按业务主键幂等去重
    # 提交:存「下一条要读的 offset」+「这条消息的时间戳」
    db.upsert(CLUSTER_ID, TOPIC, PARTITION, rec.offset + 1, rec.timestamp)

关键点:消费者始终存 last_msg_ts(方案A 切换靠它反查新集群 offset);cutover 只翻 CLUSTER_ID 配置,topic 名与代码不变;process() 必须按业务主键幂等去重,吸收边界重处理。

MM2 部署建议与 consumer group offset 同步

前面两个方案针对"offset 存在外部数据库"的 topic。但实际迁移里常有另一类 topic:用 Kafka 原生 consumer group(offset 存在 __consumer_offsets,消费者不自己存 DB)。这类 topic 不用脚本——直接让 MM2 把消费组 offset 同步到新集群即可。本节汇总 MM2 部署的关键建议 + 原生 offset 同步的开法。

版本别钉 2.8.1,用 MM2 ≥3.5

很多 MM2 资料把版本钉在 Kafka 2.8.1,这会同时埋三个雷,且都已在新版本修好:

  • offset 同步 bug:2.8.1 的 OffsetSyncStore 只存最新一条 sync,翻译不了"落后于复制进度"的消费组 → consumer 有 lag 时同步不过去。相关修复全在 3.3.3 / 3.4.1 / 3.5.0(KAFKA-14666 等)。
  • 没有 Exactly-Once:MM2 默认 at-least-once;EOS 要 Connect ≥3.3 + dedicated MM2 ≥3.5(KIP-710),且要显式开 exactly.once.source.support=enabled。2.8.1 根本没这条路。
  • 多节点扩展 bug:dedicated 模式多节点要 KIP-710(3.5.0)才可靠;2.8.1 多实例会丢配置 / 复制异常。
结论 用现代 MM2(≥3.5,建议最新 3.x),与源 broker 版本无关——一次解决 offset 同步、EOS、多节点扩展三件事。

offsetconsumer group offset 同步(原生消费组的 topic)

对于用原生 consumer group 的 topic,让 MM2 直接把消费组 offset 翻译并同步到新集群:消费者换 bootstrap.servers、group.id 不变,就从同步过来的位点续上。必须显式开两个开关(默认是关的):

# MirrorCheckpointConnector:把源消费组 offset 翻译后写入新集群 __consumer_offsets
emit.checkpoints.enabled=true            # 默认 true
sync.group.offsets.enabled=true          # 默认 false,必须手动开
sync.group.offsets.interval.seconds=5    # 同步周期
refresh.groups.enabled=true              # 尽快发现新消费组
refresh.groups.interval.seconds=5
  • 前提:MM2 ≥3.5 + IdentityReplicationPolicy(topic 同名,否则消费者得订阅 source.topic)。
  • 不用 drain 消费者:offset 同步把位点(含 backlog)带到新集群,消费者过去接着处理 backlog。
  • 会有少量重复:offset 翻译是保守(向下取整)的 → cutover 时重处理边界几条 → 消费者必须按业务主键幂等去重(at-least-once 语义)。
  • cutover 前必须验证:在新集群 kafka-consumer-groups --describe --group <g> 确认 offset 已同步且正确。否则消费者 auto.offset.reset=earliest 会从头重读、latest 会跳过 backlog(丢数据)。
混合场景:按 topic 选方式 一次迁移里两类 topic 常并存——外部 DB offset 的用前面的脚本(方案 A / B);原生 consumer group 的开 MM2 sync.group.offsets.enabled 同步。逐 topic 选对应方式即可。

形态部署形态:单机 / 容器 / 分布式

Docker Compose(单宿主)受限于一台机可复现、易起停;PoC / 固定单节点

清单部署关键配置与注意事项

  • 内部 topic 复制因子:MM2 示例常设 RF=1,但 MSK 默认 min.insync.replicas=2 → 内部 topic 写不进。把 checkpoints / heartbeats / offset-syncs.topic.replication.factor 与 offset / status / config.storage.replication.factor 设为 -1(用集群默认)或 ≥2。
  • IdentityReplicationPolicy:目标 topic 同名,迁移对生产 / 消费透明。
  • at-least-once → 幂等:MM2 重启可能产生重复,消费者必须幂等去重;要 EOS 需 ≥3.5 + 显式开 exactly.once.source.support。
  • tasks.max = min(tasks.max, 分区数),超过分区数无效;吞吐还看 max.poll.records / fetch.max.bytes / batch.size / compression。
  • 关闭自动建 topic:source.consumer.allow.auto.create.topics=false,避免删 topic 时被 consumer 自动重建。
  • 切完的 topic 从 MM2 复制列表移除,只继续复制未迁移的,避免重复 / 回环。
  • 监控:JMX → Prometheus → Grafana,看 replication latency / consumer lag / record age,判断是否追平、能否 cutover。
  • cutover 顺序:先停生产者 → 等 MM2 排空 + offset 同步落地 → 在新集群验证 offset → 切消费者 → 切生产者 → 停 MM2。

附 · MSK Tiered Storage(分层存储):应对存储增长的原地方案

当集群存储已经很大、且主要是冷数据撑起来的,Tiered Storage 是对现有 MSK 3.8 集群的原地改造——无需新建集群、无需迁移、不触碰加密配置,就能把冷数据从昂贵的 EBS 自动移到低成本托管层(S3 背书)。相比迁移到 Express broker(必须新建集群 + 数据迁移),这是"纯存储问题"的低风险解。

选型Tiered Storage vs 迁移到 Express

维度Tiered Storage迁移到 Express
是否要迁移✓原地启用,无迁移✕必须新建集群 + 迁数据
加密前提✓不要求(见下)✕本案 Replicator 被加密堵死 → 只能 MM2
解决什么存储成本 / 容量(长 retention 便宜留)存储 + 吞吐 3x + 扩容 20x + 免运维

适用痛点就是"存储大 / 贵"还要吞吐与运维升级,且接受一次迁移

纯存储问题 → Tiered Storage;要 Express 的吞吐 / 运维红利且接受迁移 → Express。

关键:Tiered Storage 不要求开 broker 间加密 存储模式(StorageMode=TIERED)与传输加密(EncryptionInTransit{InCluster, ClientBroker})是两套独立设置,互不相关。本案集群"broker 间加密关闭且不可改"堵死了 MSK Replicator(Replicator 要 IAM → 强制 broker 间加密=开),但堵不住 Tiered Storage——PLAINTEXT 集群照样能原地启用。冷层数据由 MSK 用 KMS 默认强制静态加密(at-rest),与 InCluster 开关无关;且数据走 VPC 内部、不经公网。

原理数据流转 · 三个独立时间点

数据进入分层存储有三个互相独立的触发点,别混在一起:

  • ① 复制到冷:段(segment)一关闭就立即上传,由 segment.bytes / segment.ms 决定,与 retention 无关(可能在写入后几分钟~几小时)。active 段不上传。
  • ② 从 EBS 删除:数据达到 local.retention.ms/bytes(且已确认在冷里)→ 删本地副本,冷里仍在。
  • ③ 彻底删除:数据达到 retention.ms/bytes(总保留)→ 连冷一起删。

关键:上传(早,段一关)≠ 删 EBS(晚,到 local.retention);两者之间数据在 EBS 与冷各一份。Kafka 绝不会在"确认进冷"之前删本地副本 → 不丢数据。

配置两层开关(集群级 + 每个 topic)

集群级开一次(只是"开能力"),然后每个要分层的 topic 必须单独开——开了集群 TIERED 不会自动分层任何 topic。

# ① 集群级(现有集群,一次性;3.8 满足版本要求)
aws kafka update-storage --cluster-arn <arn> \
  --current-version <v> --storage-mode TIERED

# ② 每个要分层的 topic 单独配(remote.storage.enable + 短 local.retention)
bin/kafka-configs.sh --bootstrap-server $bs --alter \
  --entity-type topics --entity-name <topic> \
  --add-config 'remote.storage.enable=true, local.retention.ms=259200000, retention.ms=604800000'
#                ↑ 本地 3 天(短)         ↑ 总 7 天
必须设短 local.retention,否则不省钱 local.retention.ms/bytes 默认 = -2(= 跟随总 retention,而 retention.bytes 默认 -1 无限)→ 数据全程在 EBS 也留一份,EBS 完全不省。要降 EBS 必须显式设一个短的 local.retention.ms 或有限的 local.retention.bytes(约束:必须 < retention)。

计费存储与计费(us-east-1)

冷层单价只比 EBS 低约 40%,真正省钱靠两点:(1) 冷层按 1 份逻辑量计费,而 EBS 按 RF×(如 3 份)物理副本计费;(2) local.retention 设短后 EBS 只扛热窗口。

冷层存量 = 总保留量,与热保留天数无关 因为段一关就上传、留到总 retention 才删,所以冷层始终装着整个 retention 窗口的全部数据(例如保留 7 天、每天 1TB ⇒ 冷层 ≈ 7TB),无论 local.retention 设 1 天还是 6 天都一样;local.retention 只决定 EBS 上额外再存几天的副本。想让冷层变小,只能调小 retention(总保留)。
成本示例(官方,省约 66%) 15 MB/s、RF=3、保留 7 天 —— 不用 TS:6× m5.large + 32.4 TB EBS ≈ $4,755/月;用 TS(本地 4 小时、总 7 天):3× m5.large + 0.8 TB EBS + 9 TB 冷层 ≈ $1,584/月,省约 66%(把全部历史读一遍,取回费仅 +$13)。

延迟冷读延迟

读冷数据由 broker 直接从冷层取出、发给消费者,不会回迁到 EBS。延迟特征是"首字节慢一下、之后接近本地":

  • 首字节(开始读冷):延迟升高(S3-GET 级,经验几十~一两百 ms,非官方 SLA)。
  • 之后顺序读:broker 预取,延迟 / 吞吐回落到接近本地 EBS。

所以顺序读一天的冷数据 ≠ 全程高延迟——只有 warmup 慢,之后流式拉满速;批量回放 / 补数完全可接受。真正会持续感到延迟的是:大量随机冷 seek、多消费者各自从老 offset 冷启动、或延迟敏感的实时消费追到了冷段。取回成本在正常消费下可忽略,只有频繁全量 replay 才需把 retrieval 费算进去。

约束要求与注意事项

  • 集群版本 ≥3.6.0(或 2.8.2.tiered),3.8 满足;仅 Provisioned 模式。
  • 不支持 kafka.t3.small —— 上线前确认 prod 实例类型。
  • 不支持 compacted topic:启用集群级 TS 要求 cleanup.policy=delete;compacted topic(changelog / KTable)不能开 TS,可按 topic 跳过。
  • 远端层最少保留 3 天;retention.ms 必填且 ≥3 天;local.retention 必须 < retention。
  • 段最小 48 MiB / 滚动最小 10 分钟;低吞吐 topic 务必调小 segment.ms,否则 active 段长期不关闭 → 数据迟迟不进冷、不下 EBS,分层形同没生效。
  • 只能按 topic 关、不能整集群关;某 topic 一旦关闭 TS 不能再开(粘性决定)。
  • 不支持 GovCloud / 中国区。
  • per-topic 配置也是优点:可只对大 / 冷 topic 开,选择性分层。
